02为什么 grep 根本不够用？
很多人会说：「那让 Agent 多 grep 几次不就行了？」问题在于，grep 只能做字符串匹配，它完全不理解代码语义。下面四种常见场景，grep 全都抓瞎：
🔀
动态分发
接口方法调用在编译期不知道具体实现类。handler.handle(req)——grep 搜 "handle" 找不到它实际调的是哪个实现。多态、依赖注入、策略模式都会让 grep 失效。
🗺️
框架路由
NestJS 的@Post('users')装饰器定义了 URL → Controller 方法的映射。grep 搜 "/users" 找不到这个关系——URL 路径是通过框架约定拼接的。
📦
再导出链
模块 A 从模块 B 导入，B 又从 C 再导出。grep 搜函数名只搜到 B 的再导出语句，找不到 C 里的原始定义。跨多个 barrel 文件的跟踪对 grep 是噩梦。
🌐
跨语言桥接
React Native 中 JS 调用原生模块，Swift 调用 ObjC——grep 只能搜索一种语言的字符串，跨越语言边界的调用关系完全不可见。
📌 核心问题
grep 给你一堆匹配的字符串位置，AI Agent 真正需要的是一张可遍历的关系图。从这个函数能走到哪些函数？改了这里会影响哪里？谁调用了这个接口？——这些都是图查询，不是文本搜索。
03Structure Discovery Overhead
这个问题有个正式的名字：Structure Discovery Overhead——Agent 在每次查询时都从零开始，重新发现代码的拓扑结构。
具体来说，Agent 每次都在重复回答这些问题：
❓这个文件里有哪些函数和类？
❓这个函数调用了哪些其他函数？
❓这个类继承了谁？实现了哪些接口？
❓这个 URL 路由注册在哪个文件里？
❓改了这里，哪些文件会受影响？
这些信息在不同查询之间几乎不变——你的代码在你写完之后，调用关系就确定了。但 Agent 每次都要重新发现一遍。

→ 看看是不是你要的

无论你来过这个图书馆多少次
VS
✅ 有卡片目录
查卡片目录找到位置
→ 记下书架编号
→ 径直走过去拿书
→ 一步到位
找一本书要 30 秒
因为结构信息已经建好了
📌 关键点：目录系统是一次建好、反复使用的。图书馆不是你每次进门才给书分类——分类在你进门之前就做好了。CodeGraph 做的事情完全一样：在你开始提问之前，把代码的「卡片目录」建好。
05CodeGraph 就是那张卡片目录
CodeGraph 是一个预索引的代码知识图谱。它读取你的代码库，提取所有函数、类、方法、接口，以及它们之间的关系，存进一个本地图数据库。
N
Node（节点）
函数、类、方法、接口、常量、路由处理器、Vuex mutation、Pinia store——每一个可识别的代码实体。
E
Edge（边）
调用、导入、继承、实现、路由映射、跨语言桥接——节点之间的关系。
G
Graph（图）
节点 + 边的整体拓扑结构。不是列表，不是表格，是一个可遍历的有向图。
有了这张图，Agent 的问题变了：

✅ 使用 CodeGraph

06CodeGraph 不是什么
理解一个工具是什么很重要，理解它不是什么同样重要：
✕
不是代码搜索引擎
它是结构图，不只是文本索引。它理解调用关系、继承链、框架路由，而不是简单的关键词匹配。
✕
不是云端服务
100% 本地运行，数据从不离开你的机器。没有云上传、不需要 API Key、不依赖任何外部服务。
✕
不是静态分析工具
不检查 bug，不替代 ESLint 或 Sonar。它告诉你「结构是这样的」，而不是「结构有问题」。
✕
不是替代 AI 推理
推理仍然是 Claude 等 AI 模型完成。分工明确：CodeGraph 管「代码在哪里」，Agent 管「代码怎么办」。
