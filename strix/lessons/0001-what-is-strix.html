01一句话说清楚 Strix
Strix 就是一个 AI 黑客团队，你给它一个目标（代码目录、GitHub 仓库、或者网址），它自己启动一堆 Agent：有的负责侦察、有的负责找漏洞、有的负责写 PoC 验证、有的负责修。每个 Agent 都有自己的工具箱——HTTP 代理、浏览器、终端、Python 运行时——就跟真人黑客似的。
1
你给目标
本地代码目录 / GitHub 仓库 / 线上网址
2
AI Agent 团队干活
侦察 → 代码分析 → 漏洞定位 → PoC 验证 → 修复建议，Agent 团队协作
3
收到可操作的报告（甚至修复 PR）
不是 PDF 废话，是能直接合并的修复 PR
02能干啥——能扫出来的漏洞类型
Strix 覆盖的漏洞类型相当全，基本覆盖了 OWASP Top 10：
🔐 权限控制
IDOR、提权、鉴权绕过
💉 注入攻击
SQL、NoSQL、命令注入
🕸️ 服务端
SSRF、XXE、反序列化
📄 客户端
XSS、原型污染、DOM 攻击
🧩 业务逻辑
条件竞争、逻辑漏洞
🔑 认证相关
JWT、Session 管理
关键的区别是：Strix 不是纯静态分析。它真的会跑你的代码，试 payload，看能不能打进去。不像 SAST 工具报警 "这里可能有 SQL 注入"，Strix 是直接往你数据库插一条试试——能在响应的数据里看到自己插的东西才算数。误报率低得多。
03安装——三种方式，看哪个顺手
先确认你有了啥：Docker（必须，Strix 在沙箱里跑）、Python 3.10+、一个 LLM API Key（OpenAI / Anthropic / Google 的都行）。
方式 1：一行命令（推荐）
curl -sSL https://strix.ai/install | bash
自动检测系统，装到全局。最简单，不用折腾。
方式 2：pipx（不污染全局 Python）
pipx install strix-agent
如果你已经在用 pipx 管理工具（像 poetry、black 之类的），推荐这个。隔离干净，不影响其他项目。
方式 3：源码安装
git clone https://github.com/usestrix/strix.git
cd strix
pip install -e .
适合想改源码或者研究内部实现的人。
⚠️ 踩坑提示
• 第一次安装后运行 strix --version 确认装上了。如果报 "command not found"，可能是 PATH 没刷——关掉终端重开。
• 第一次运行 strix 时，会自动 pull Docker 沙箱镜像（大概几百 MB），需要联网且等几分钟。别以为卡住了。
• Docker 必须是 running 状态。跑 docker ps 确认一下。
04配置——设好 API Key 就能跑
Strix 背后是大模型在驱动，所以你得配一个 LLM 的 API Key。支持 OpenAI、Anthropic、Google，甚至本地的 Ollama/LMStudio。
环境变量配置方式：
# 选一个你用的模型
export STRIX_LLM="openai/gpt-5.4"
export STRIX_LLM="anthropic/claude-sonnet-4-6"
export LLM_API_KEY="sk-your-key-here"
如果用本地模型（Ollama）：
export STRIX_LLM="ollama/llama3"
export LLM_API_BASE="http://localhost:11434"
💡 小贴士
第一次运行后，Strix 会自动把配置存到 ~/.strix/cli-config.json。后续不用每次设环境变量了。如果以后想换模型，直接改这个文件也行。
05第一次扫描——跑起来看看
激动人心的时候到了。配好 API Key 之后，随便找个项目试试：
扫本地代码：
strix --target ./my-app

# 或者指定扫描模式
strix --target ./my-app --scan-mode quick
第一次跑会先 pull Docker 沙箱镜像（大概一两分钟），然后扫码开始。之后的结果存到 strix_runs/ 目录下。
扫 GitHub 仓库：
strix --target https://github.com/some/project
黑盒扫线上网站：
strix --target https://your-app.com
⏱ 运行多久？
取决于项目大小和扫描模式：quick 模式几分钟到十几分钟；standard 模式可能半小时；deep 模式几小时甚至更久。这期间 Strix 会打印 Agent 在干啥，你可以泡杯咖啡看着它干活。按 Ctrl+C 可以中断，已经找到的漏洞不会丢。
06三种扫描模式，啥时候用哪个
模式命令什么时候用
⚡ Quick--scan-mode quickCI/CD 里跑，每次 PR 触发，只看严重漏洞。几分钟搞定。
🔍 Standard默认（不用加）日常开发中定期跑，覆盖面适中。半小时左右。
🐢 Deep--scan-mode deep上线前全量审计，或者重大版本发布前。几小时起步。
07CI/CD 集成——每次 PR 自动扫
这是 Strix 最实用的场景——每次提 PR 自动扫一遍，有漏洞就阻止合并。GitHub Actions 配置长这样：
name: strix-penetration-test
on:
pull_request:
jobs:
security-scan:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v6
with:
fetch-depth: 0
- name: Install Strix
run: curl -sSL https://strix.ai/install | bash
- name: Run Strix
env:
STRIX_LLM: ${{ secrets.STRIX_LLM }}
LLM_API_KEY: ${{ secrets.LLM_API_KEY }}
run: strix -n -t ./ --scan-mode quick
注意 -n 参数表示非交互模式，适合 CI。API Key 要放到 GitHub 的 Secrets 里，别硬编码。
💰 费用提醒
每次扫描都会消耗 API token，根据项目大小和扫描模式，一次 CI 跑的 LLM 费用可能在 $0.5-$5 之间。quick 模式最省。如果你用本地模型（Ollama），完全免费但会慢一些。
⚠️风险和局限性
🔴
它是真黑客——别在生产环境乱跑
Strix 是真的会发 HTTP 请求、试 payload、往数据库插数据的。在线上生产环境运行可能会导致数据污染或服务异常。建议在 staging 或本地环境跑。
🟡
AI 不是万能的
复杂业务逻辑漏洞、需要深度领域知识的漏洞，AI 可能发现不了。Strix 是个好助手，不是安全团队的替代品。
🟡
依赖 Docker 和网络
沙箱跑在 Docker 里，没 Docker 跑不了。而且每次首次 pull 镜像挺大的（几百 MB）。
