"给我一段视频，我还你一个完整 3D 场景。"这是 LingBot-Map 的承诺——不用训练、不用优化、不用 GPU 集群，单卡跑 20 帧/秒，重建百万帧级别的长视频。
你拿手机拍了一圈办公室，或者从无人机上录了一段飞行视频。回头想把这些片段变成一个你可以在任意角度漫游的 3D 场景——这事儿以前怎么干的？
传统的办法：COLMAP、NeRF 那一套。拍完照片，跑几十个小时的特征匹配和光束法平差。要是视频长度超过几百帧，内存先炸了。想换几个视角看看？等几小时渲染吧。这是 3D 视觉领域多少年的痛——能做，但太慢了，太麻烦了，没法用。
好家伙，转折来了。
2026 年 4 月，一个叫 Robbyant 的团队在 arXiv 上丢了一篇论文，同时开源了代码。项目名 LingBot-Map，上线没几天 GitHub Star 直接冲着 10K 去了。啥来头？前馈式、流式、20 FPS、万帧以上场景直接重建——你把视频丢进去，它一边看一边搭 3D 场景，根本不需要先跑一轮优化。

01这事儿难在哪？LingBot-Map 怎么搞定的？
你琢磨一下，"从视频重建 3D 场景"这个问题到底难在哪。
第一，视频是一帧一帧的，你得弄清楚每张图片里什么东西在什么位置——深度估计。第二，相邻两帧画面是连续的，你需要知道摄像机怎么动——位姿估计。第三，拍了一大圈回到原点的时候，你会发现开头和结尾的位姿对不上了——漂移校正（loop closure）。第四，也是最重要的——所有这些都得在线完成，不能先把所有帧都存下来再统一优化，不然长视频就是死路一条。
LingBot-Map 的做法，说白了就是设计了一个叫 Geometric Context Transformer（几何上下文 Transformer）的架构。这玩意儿一次把上面四个问题全包了：
1
Anchor Context（地锚上下文）
把第一帧作为"锚点"，所有后续帧都跟锚点建立坐标关联。这就解决了"第一帧坐标系"的问题。
2
Pose-Reference Window（位姿参考窗口）
处理每一帧的时候，不仅看当前帧，还参考前后邻近帧的信息。这样相邻帧之间的位姿变化就平滑了，深度估计也更准确。
3
Trajectory Memory（轨迹记忆）
用一个 paged KV cache（分页键值缓存）来存整条视频的上下文信息。每来一帧，只需要更新 cache，不需要重算前面所有帧。这就是能跑万帧以上视频的秘密武器。
说白了，前面那些传统方法需要"先攒后算"，LingBot-Map 是"边看边建"——每来一帧，Transformer 用已经积累的上下文信息做预测，预测完把这一帧的信息压缩进 cache，然后处理下一帧。整个过程是前馈的（feed-forward），不需要迭代优化。
💡 小白解释
传统方法像拼拼图——把所有碎片倒出来，先分类，再比对边缘，最后一块一块拼。LingBot-Map 像看地图走路——每走一步都知道自己在哪，前面是什么。所以它快得多，而且不怕路长。
02实测数据——比同类好多少？
论文里做了大量 benchmark 对比，涵盖了室内（TUM、7-Scenes、NRGBD）、室外（KITTI、Oxford Spires）、大规模场景（Tanks and Temples、ETH3D）等各种数据集。咱们说几个重点结论：
🏆
位姿估计精度（ATE RMSE）
在 KITTI 数据集上，LingBot-Map 的绝对轨迹误差显著低于所有现有流式方法——包括 DPV-SLAM、Droid-SLAM、Deep-Patch 等。也就是说，它不光快，还更准。
🏗️
3D 重建质量（F-Score）
在 Tanks and Temples 和 ETH3D 上，LingBot-Map 的重建 F-Score 超过了原始 VGGT 和 DUSt3R，甚至在某些场景接近甚至超越了专注于小场景的 DUSt3R 优化版本。
⚡
推理速度
在 518×378 分辨率下达到约 20 FPS，且支持超过 10,000 帧的长序列。开启 --compile 和 FlashInfer 后端后，速度还能进一步提升。
有个特别精彩的数据：在室内长时间漫游约 25,000 帧、13 分钟的视频上，LingBot-Map 用窗口推理模式跑完了全序列，最终重建结果清晰稳定，没出现位姿崩坏。你想想，25,000 帧放在传统方法上，光做特征匹配就得等多久。
03怎么用？一条命令感受一下
安装其实不难，但这玩意儿目前只支持 Linux + CUDA。如果你有台带 NVIDIA 显卡的 Ubuntu/Debian 机器，跟着走就行。
# 1. 创建 conda 环境
conda create -n lingbot-map python=3.10 -y
conda activate lingbot-map

# 2. 安装 PyTorch (CUDA 12.8)
pip install torch==2.8.0 torchvision==0.23.0 --index-url https://download.pytorch.org/whl/cu128

# 3. 安装 lingbot-map
git clone https://github.com/Robbyant/lingbot-map.git
cd lingbot-map
pip install -e .

# 4. 安装 FlashInfer（强烈推荐）
pip install flashinfer-python
⚠️ 新手最容易踩的几个坑：
• PyTorch 2.8.0 是推荐版本，因为 NVIDIA Kaolin（批渲染需要）有这个版本的预编译 wheel。如果你只用交互式 demo，可以用更新的 PyTorch，但批渲染就得从源码编译 Kaolin 了
• FlashInfer 是纯 Python 包，首次使用时 JIT 编译 CUDA 内核，所以第一次跑需要等一会儿
• 如果不方便装 FlashInfer，可以用 --use_sdpa 退回到 PyTorch 原生的 SDPA 注意力机制
• 天空分割需要 onnxruntime，首次运行会自动下载分割模型
安装完模型之后，跑个样例看看效果：
# 下载模型（从 HuggingFace 或 ModelScope）
huggingface-cli download robbyant/lingbot-map --local-dir ./checkpoints

# Courthouse 场景
python demo.py --model_path ./checkpoints/lingbot-map-long.pt \
    --image_folder example/courthouse --mask_sky

# 浏览器打开 http://localhost:8080 查看 3D 场景
这个命令启动了交互式 viser 查看器，你可以在浏览器里用鼠标旋转、缩放、平移，从任意角度观察重建的 3D 点云。不用你训练，不用调参，一条命令出结果。

04这些隐藏技巧，你一定要知道
LingBot-Map 的 README 写得很详细，但有几个点值得单独拿出来讲。这些是真正上手的时候会遇到的关键问题。

🎯 Keyframe Interval——长视频的关键开关

模型在 320 帧上训练（Video RoPE 编码），所以当 KV cache 里存了超过 320 帧时，性能会开始下降。怎么办？每 N 帧只存一帧作为"关键帧"，其他帧做预测但不进 cache。这样实际有效序列长度 = 缓存帧数 × keyframe_interval，一下子就把容量翻了 N 倍。
# 每 2 帧存一个关键帧，有效长度翻倍
python demo.py --model_path /path/to/model.pt --image_folder /path/to/images \
    --keyframe_interval 2 --camera_num_iterations 4

🪟 Windowed Mode——上万帧视频的救星

超过 3000 帧的极长视频，就得用滑窗模式了。窗口大小 128（是指 KV cache 槽位，不是实际帧数），每 128 个槽位中前 8 个给 scale frame，剩下 120 个给关键帧。如果 keyframe_interval=13，那一个窗口就覆盖 8 + 120×13 = 1568 帧。相邻窗口之间共享 8 个关键帧作为重叠，保证跨窗口位姿连续。
python demo.py --model_path /path/to/model.pt --video_path video.mp4 --fps 10 \
    --mode windowed --window_size 128 --overlap_keyframes 16 \
    --keyframe_interval 2

☁️ Sky Masking——室外场景的魔法

室外重建最大的麻烦是天空。天空没有深度，但点云会把蓝天沿着锥体投射到无限远，导致重建结果里飘着一堆噪声点。LingBot-Map 集成了 ONNX 天空分割模型，自动识别并过滤天空区域。首次使用会自动下载分割模型，分割结果会缓存到磁盘，后续运行直接复用。

💾 显存不够怎么办？

LingBot-Map 默认已经把逐帧预测结果 offload 到 CPU 了（--offload_to_cpu 默认开启）。如果还是 OOM，可以缩减双向尺度帧数量：
# 默认 num_scale_frames=8，改成 2 省显存
python demo.py --model_path /path/to/model.pt --image_folder /path/to/images \
    --num_scale_frames 2
05进阶玩法：渲染电影级点云漫游
交互式查看器虽然好用，但你要想输出一段炫酷的 3D 点云漫游视频，就得用离线渲染管线了。demo_render/batch_demo.py 是离线版的入口，输入视频或图片序列，直接输出一个 MP4 漫游视频。
最骚的是它的摄像机路径可以通过 YAML 文件自定义——你可以让摄像机"跟随"原始轨迹，也可以做"鸟瞰"视角，甚至把两种模式组合在同一个视频里分段使用。想一想：一段 13 分钟的室内漫游，先追随视角走一遍，最后拉升到鸟瞰，全景一览无余——绝对是论文展示和项目 demo 的大杀器。
🎬 离线渲染命令（25,000 帧室内漫游）
python demo_render/batch_demo.py \
  --video_path indoor_travel.MP4 \
  --output_folder /data/outputs/indoor_travel/ \
  --model_path /path/to/lingbot-map.pt \
  --config demo_render/config/indoor.yaml \
  --mode windowed --window_size 128 \
  --keyframe_interval 13 --overlap_keyframes 8 \
  --camera_vis default --keyframes_only_points \
  --frame_tag --frame_tag_position top_right
输出文件：pointcloud.mp4（点云漫游）+ pointcloud_rgb.mp4（原始 RGB）+ 可配置的 YAML 摄像机路径
更绝的是，这套流程连 AI 生成的视频都能重建——LingBot-Map 拿自己家"LingBot-World"世界模型生成的视频做输入，一样能重建出 3D 场景。生成式 AI + 3D 重建闭环了属于是。
06说了这么多，这玩意儿能干嘛？
聊点实在的——这种技术能用在什么地方？
🏠
室内 3D 扫描
拿着手机/手持云台走一圈，自动生成室内 3D 模型。房产展示、室内设计、数字孪生，全都能用。以前做一套房子得几千块请人扫描，现在一条命令的事。
🚁
无人机航测
无人机飞一圈录视频，下来跑个 LingBot-Map，就能拿到 3D 正射影像和地形模型。20 FPS 的处理速度意味着落地前基本就算完了。
🚗
自动驾驶
车载摄像头流式重建周围环境。KITTI benchmark 上的表现说明它完全适合自动驾驶场景的实时位姿估计和语义建图。
🎮
游戏和影视
实景扫描生成 3D 资产，快速搭建游戏场景或影视前期预览。Apache 2.0 协议，商用无压力。
