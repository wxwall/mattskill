01一句话说清楚 /goal
/goal 就是一个"自动挡"开关。你给它一个可验证的完成条件，Claude 就自己一轮接一轮地干活——改代码、跑测试、看结果、再改——直到条件满足，自动停下。你不用再当监工，每轮结束后等它主动开下一轮。
1
你设一个目标条件
"/goal npm test 全部通过"
2
Claude 自动循环干活
每轮结束 → 轻量评估器检查条件 → 没达成？开下一轮
3
条件满足，自动停，你验收
◎ /goal active 指示器消失，控制权回到你手上
02核心机制——循环 + 评估器
/goal 的引擎是两个东西打配合：一个自动循环、一个轻量评估器。
🔄 自动循环（每轮结束 → 自动开下一轮）
以前是：你发话 → Claude 干活 → 干完了等你再发话。现在你只发话一次，Claude 干完一轮自己判断"还要不要继续"——要就接着干，不要就停。
🧠 轻量评估器（默认 Claude Haiku）
每轮结束后，一个极低成本的小模型（Haiku）扫一眼对话记录，判断"你说的条件现在满足了吗？"——满足就停，不满足就继续。它只看对话里已经有的内容，不会自己偷偷跑命令或读文件。这意味着你的条件必须是 Claude 能在对话中证明的，比如"测试退出码为 0"，而不是"代码好看多了"。
/goal 运行流程
你输入 /goal 「条件」
↓
Claude 干一轮活（可能很多步工具调用）
↓
Haiku 评估器扫一眼对话记录
↙
条件满足 ✓
↓
自动停止，控制权归还
↘
条件不满足 ✗
↓
自动开下一轮，继续干
03条件怎么写——这是最关键的事
用 /goal 唯一需要费脑子的地方就是写条件。条件写得好，Claude 干得漂亮、自动收工；条件写得烂，要么永远停不下来，要么跑偏了十万八千里。
好条件长啥样？说白了就三个要素：
📏
可测量的终点
不是你感觉"好了"，而是能被验证的东西——测试 exit 0、文件存在、curl 返回 200
🔍
能在对话中证明
评估器只看对话记录——它不会自己去跑命令。条件必须是对话里已经呈现了证据的事
🚧
有边界约束
"别碰 auth 目录下任何文件"、"只改 test/ 里的东西"——防止 Claude 跑偏
✅ 好条件 vs ❌ 烂条件
✅ 好条件示例
"/goal 修复所有失败的测试，直到 npm test 退出码为 0"
"/goal 把 src/utils.ts 拆成最多 5 个文件，每个不超过 200 行，且 npm test 通过"
"/goal 处理 GitHub 上所有 high-priority 标签的 issue，直到我确认没有剩余"
❌ 烂条件示例（别这么写）
"/goal 优化一下项目" —— 啥叫优化？永远没个头
"/goal 让 UI 好看点" —— 主观判断，评估器没法判断
"/goal 重构一下代码" —— 太模糊，Claude 会乱跑
📌 万能模板
/goal [干什么活] 直到 [可测量的终点] 且不 [约束条件]
比如：/goal 修复 test/auth/ 下所有失败用例，直到 npm test -- test/auth/ 退出码为 0，且不改 src/ 下任何文件
04实际操作——怎么用 /goal
用法贼简单，就几个命令：
🎯 设目标
/goal npm test 全部通过
📊 查看状态
/goal （不带参数）—— 显示当前条件、跑了多少轮、花了多少 token、评估器的最新判断
⏹ 取消
/goal clear / /goal stop / /goal cancel —— 都是一回事
◎ 目标激活时你会看到
屏幕上方出现 ◎ /goal active 指示器，显示：已用时间、轮次数、累计 token 数。如果 token 烧得太快但没进展，按 Esc 或 /goal clear 打断它。
05什么时候该用？什么时候别用？
/goal 不是万能的。用对了爽到飞起，用错了反而浪费时间。记住这些判断：
✅ 适合用 /goal 的场景
• 修测试——"跑到所有测试通过为止"（最经典）
• 批量迁移——"把所有 class 组件改成函数组件"
• 生成文件——"CHANGELOG.md 包含本周所有合并的 PR 条目"
• CI 流水线——用 claude -p "/goal ..." 跑完自动退出
❌ 不适合用 /goal 的场景
• "让 UI 好看点"——主观判断，评估器判断不了
• 一次性小编辑——30 秒能搞定的活，/goal 的循环开销比活还大
• 架构决策——每个分叉都需要你把关
• 第一次碰一个项目——你还没给 Claude 足够的领域上下文
• 破坏性操作可能难以撤销——/goal 每轮都能调危险工具不用你确认
🔥 黄金组合：/goal + Auto Mode
Auto mode 去掉的是每步工具调用的确认——你不用对每个 bash 命令点头。/goal 去掉的是每轮对话的确认——你不用在每轮结束时说"继续"。两个一起开，Claude 就是真正的自动驾驶了。
06实战套路——老手都这么玩
用多了你会发现，最好的用法不是一个裸 /goal 甩上去——而是跟 /plan 打配合：
📋 套路 1：先 Plan 后 Goal（最推荐）
先 /plan 让 Claude 设计方案，你看完觉得 OK 了，把验收标准直接变 /goal 的条件。Plan 锁策略，Goal 锁执行。
📝 套路 2：让条件里要求维护 progress.md
在条件里加一句"维护 progress.md 记录每轮干了啥"——这样你去喝咖啡回来，能看到完整审计记录。
🧪 套路 3：CI 里用 -p 非交互模式
claude -p "/goal CHANGELOG.md 包含本周所有合并的 PR 条目" —— 跑完自动停。建议外面套个超时。
💰 套路 4：加轮次上限保底
条件后加一句"或最多 20 轮后停止"——兜底保险。尤其适合你第一次试用、不确定会不会跑飞的时候。
🔀跟 /loop 、stop hook 的区别
Claude Code 有好几个"自动"类功能，很容易搞混。一张表讲清楚：
