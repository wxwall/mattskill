01为什么 AI 需要"看懂"代码
当你让 AI 编程助手（Claude Code、Cursor 等）修改一个函数时，它当前的做法是：先搜索文件，然后逐个读取，再理解上下文。在大项目中，这就像"在图书馆里一本一本翻书找一句话"——消耗大量 token 和时间。
❌ 没有 CodeGraph
🔍搜索文件名 → 猜测哪个文件包含目标代码
📖逐个读取文件 → 消耗大量 token
🤔推断函数调用关系 → 可能理解错误
🔄多轮工具调用 → 响应慢、token 消耗高
✅ 使用 CodeGraph
⚡一次 MCP 查询 → 获取入口点 + 相关符号 + 调用流
📊知识图谱直接回答 → 零文件读取
🎯精确的调用链和影响范围 → 修改有据可依
💰工具调用减少 58% → token 消耗降低 47%
📌 核心洞察：CodeGraph 解决的不是"代码分析"问题，而是AI Agent 的上下文获取效率问题。在大项目中，代码搜索和读取占了 Agent 大部分 token 消耗，CodeGraph 让这部分成本趋近于零。
02什么是 CodeGraph
CodeGraph 是一个本地优先的代码智能工具，既是一个 CLI 命令，也是一个 MCP Server。它解析你的源码、构建知识图谱，并通过标准协议暴露给 AI Agent。
1
🔒 100% 本地运行
数据从不离开你的机器，无需 API Key，没有外部服务。你的代码只有你知道。
2
🌳 tree-sitter 解析引擎
支持 20+ 种编程语言（TypeScript、Python、Go、Rust、Java 等），精确解析 AST 而非字符串匹配。
3
🗃️ SQLite 图存储
用 SQLite + FTS5 全文搜索引擎存储符号、边和文件，零依赖、零配置。
4
🔌 MCP 协议接入
通过 Model Context Protocol 标准协议与 Claude Code、Cursor 等 AI Agent 通信，即插即用。
03四阶段流水线
CodeGraph 的核心是一个四阶段流水线，把源码变成 AI Agent 可用的知识图谱：
阶段 1
Extract
tree-sitter 解析
↓
阶段 2
Store
SQLite 存储
↓
阶段 3
Resolve
解析引擎
↓
阶段 4
Serve
MCP Server
📤
1. Extract — 解析源码
tree-sitter 读取每个文件，生成 AST，从中提取符号（函数、类、方法、接口、常量）和关系（调用、导入、继承）。
💾
2. Store — 存入 SQLite
提取的符号和关系写入 .codegraph/codegraph.db。FTS5 全文索引支持模糊搜索。
🔗
3. Resolve — 解析引用关系
构建调用图、继承链、框架路由映射（识别 17 种框架的路由定义）和跨语言桥接。
🌐
4. Serve — 通过 MCP 提供服务
启动 MCP Server，暴露 codegraph_explore 等工具。AI Agent 一次查询即可获取入口点、相关符号、调用流和影响范围。
