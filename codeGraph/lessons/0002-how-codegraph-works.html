第 2 课 · 原理

CodeGraph 的四阶段流水线

📚 课程 0002 ⏱ 预计 15 分钟 ← 第 1 课
📖 回顾第 1 课：AI Agent 每次查询都要从零 grep/read 来发现代码结构——这叫 Structure Discovery Overhead。CodeGraph 的思路是提前把代码库变成一张可查询的图。

1. 核心数据结构：Node、Edge、Graph

CodeGraph 的数据模型很简单——三种东西组合在一起：

🔵
Node · 节点

代码中的可识别实体。
函数、类、方法、接口、常量、路由处理器、Vuex mutation、Pinia store、Celery task……
一个 Node = 一个"可以被调用、被引用、被继承"的东西

➡️
Edge · 边

节点之间的有向关系。
A 调用 B · A 导入 B · A 继承 B · URL → Handler · Swift ↔ ObjC
Edge 是有方向的——从调用者指向被调用者

🗂️
Graph · 图

所有 Node + 所有 Edge = 一张有向图。
不是列表，不是表格，是可遍历的关系网络。Agent 做的是图遍历。

有了这种数据结构，Agent 的问题就变成了图上的导航问题——"从这条路出发，沿着调用关系走下去，经过哪些节点，最终到达哪里？"

3. 阶段一：Extract（提取）

🔧 tree-sitter

输入：源码文件（24 种语言）
输出：AST → 符号列表（函数、类、方法……）+ 关系列表（调用、导入、继承……）

CodeGraph 使用 tree-sitter 作为语法解析引擎。tree-sitter 有几个关键特性让它特别适合这个场景：

🌐 语言无关

tree-sitter 通过语法定义文件（grammar files）来支持不同语言，不需要每种语言写一套解析器。所以 CodeGraph 能支持 24 种语言而无需维护 24 套解析代码。

⚡ 增量解析

tree-sitter 可以只重解析修改过的代码区域，而不是每次重解析整个文件。这比编译器级别的解析快得多——不需要类型检查、不需要语义分析。

🛡️ 容错性强

代码在编写过程中经常是不完整的（缺少分号、括号未闭合）。tree-sitter 的错误恢复能力让它能从不完整代码中依然提取有效信息。

Extract 阶段做的事可以理解为：把文本文件变成结构化数据。比如这段 TypeScript 代码： 

class UserService {
  constructor(private repo: UserRepository) {}
  async create(data: CreateUserDto): Promise<User> {
    return this.repo.save(data);
  }
}

tree-sitter 解析后，CodeGraph 从中提取：

Node: UserService (class)
Node: UserService.create (method)
Node: UserRepository.save (method)
Edge: UserService.create → calls → UserRepository.save
Edge: UserService → depends on → UserRepository

4. 阶段二：Store（存储）

🗄️ SQLite + FTS5

输入：提取出的符号和关系
输出：.codegraph/codegraph.db

Extract 阶段产出的符号和关系，被写入一个本地 SQLite 数据库。选 SQLite 有几个明确的考量：

🏠 零配置嵌入式

不需要安装数据库服务、不需要配置连接字符串、不需要守护进程。建好文件就能用。对于开发工具来说这是最好的用户体验。

🔍 FTS5 全文搜索

SQLite 内置的 FTS5 全文搜索引擎 支持布尔查询、前缀搜索、短语匹配。比 LIKE '%keyword%' 快几个数量级。

🔒 本地安全

数据 100% 留在你的硬盘上。没有网络通信、没有 API Key、没有第三方服务。这对于企业代码库来说是不可妥协的要求。

5. 阶段三：Resolve（解析关联）

🧩 解析引擎

输入：符号引用（符号 A 可能引用了符号 B）+ 导入关系
输出：确定性的调用图 + 继承链 + 框架路由映射 + 跨语言桥接

这是整个流水线中最复杂的阶段。Extract 阶段只知道"这个位置调用了一个叫 save 的方法"，但不清楚这个 save 到底是谁。Resolve 负责回答：

  • 跨文件：this.repo 的声明在构造函数参数里，类型是 UserRepository——去 UserRepository 的定义里找 save 方法。
  • 跨模块：如果 A 从 B 导入，B 又从 C 再导出，需要追踪到 C 的原始定义。
  • 多态分发：接口方法调用需要找到所有实现该接口的类中对应的方法。
  • 框架约定：@Post('users') 装饰器 → 对应的 URL 路径从 Controller 级别的 @Controller('users') 拼接得到。

CodeGraph 的 Resolve 阶段还能处理框架感知的路由解析。它识别的框架模式包括：

Django URLconf
Flask @app.route
FastAPI @app.get
Express Router
NestJS @Controller
Laravel Route::get
Rails resources
Spring @RequestMapping
Gin router.GET
React Router Route
Vue Router
ASP.NET [Route]
Axum .route()
… 17 种框架

v1.1.0 还新增了跨语言桥接：Swift ↔ ObjC 的自动桥接、React Native 的 RCT_EXPORT_METHOD 桥、TurboModules、Expo Modules DSL。每条桥接边标记为 provenance: heuristic，提醒 Agent 这是推断出来的边。

6. 阶段四：Serve（服务）

🖥️ MCP Server

输入：Agent 的结构化查询（通过 MCP 协议）
输出：源码片段 + 完整调用链 + 变更影响范围分析

CodeGraph 通过 MCP（Model Context Protocol） 与 AI Agent 通信。MCP 是 Anthropic 发布的一个开放协议，定义了 AI 模型如何与外部工具交互。CodeGraph 启动一个本地的 MCP Server，Agent（如 Claude Code）在需要时自动启动它并调用其工具。

CodeGraph 暴露的工具
工具名 功能 默认
codegraph_explore 主入口：一次调用返回入口点、调用流、影响范围 ✅ 可见
codegraph_node 获取单个符号详情、读取完整源文件 ❌ 隐藏
codegraph_search FTS5 全文搜索 ❌ 隐藏
codegraph_callers 谁调用了这个符号 ❌ 隐藏
codegraph_callees 这个符号调用了谁 ❌ 隐藏

设计哲学：只暴露一个主工具 codegraph_explore，让它足够强大来回答大多数问题。隐藏工具可通过 CODEGRAPH_MCP_TOOLS 环境变量重新启用。

自动同步机制

CodeGraph 不是一次性建完就完事了——你的代码会变，图也要跟着变。它有三层保证：

👁️ 文件监听

OS 原生事件（macOS FSEvents / Linux inotify / Windows ReadDirectoryChangesW）→ debounce（默认 2 秒静默窗口）→ 增量同步。写完代码保存，图自动更新。

⚠️ 过期警告

debounce 窗口期间，刚编辑的文件可能还未同步。CodeGraph 返回数据时会附上 staleness banner，告诉 Agent "这个文件刚改过，图可能稍滞后"。

🔁 重连校验

Agent 重启后（或在新会话中），MCP Server 重连时用 (size, mtime) + 内容哈希快速对比，只索引有变化的文件。
