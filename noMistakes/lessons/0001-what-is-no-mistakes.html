01为什么需要 no-mistakes
AI 编程助手让代码产出速度大幅提升，但带来的问题是——代码质量参差不齐，PR 里经常混入"slop"（注水代码）。传统的 CI/CD 只能发现你已经知道的错误，而 no-mistakes 在代码到达 CI 之前就做 AI 审查。
❌ 没有 no-mistakes
✍️AI 生成代码 → 直接提交推送
🔍人工 Review 发现低级错误
🔄来回修改 → 时间浪费
😤"又是 AI 写的垃圾代码"
✅ 使用 no-mistakes
🤖AI 生成代码 → 推送到 no-mistakes 代理
⚡AI 自动审查：review → test → docs → lint
🛠️机械问题自动修复，意图问题等你确认
🎯只有通过全部检查，才推送并打开 PR
02什么是 no-mistakes
no-mistakes 是一个用 Go 编写的本地 Git 代理。它位于你的本地仓库和远程仓库之间，拦截 git push 命令，在独立的 disposable worktree 中运行 AI 验证流水线，只有通过全部检查后才真正推送并创建 PR。
1
🔌 本地 Git 代理
不修改 Git 本身。它注册一个新的 remote，拦截 push 到代理仓库，运行验证流水线，通过后转发到真正的远程仓库。
2
🧹 Disposable Worktree
每次触发在独立的临时工作目录中运行流水线，不干扰你当前的工作。检查完成后自动清理。
3
🤖 Agent 无关
支持 Claude、Codex、RovoDev、OpenCode、Pi、Copilot 等任意 AI Agent。通过 acp:<target> 可扩展。
4
🧑‍⚖️ 人类始终掌握决定权
机械修复（格式化、lint）自动应用；涉及意图的改动等待你审批。你决定什么能通过。
03核心架构：AI 验证流水线
no-mistakes 的核心是一条七阶段流水线。当代码推送到代理仓库时，流水线在 disposable worktree 中依次执行：
1
review
AI 代码审查
↓
2
test
运行测试
↓
3
docs
文档检查
↓
4
lint
代码规范检查
↓
5
push
转发到远程
↓
6
PR
自动创建 PR
↓
7
CI
监控 CI 并修复
📌 核心思想：真正的质量保障不止于 lint 和测试。AI 审查可以发现逻辑错误、安全隐患、设计问题——这些是传统工具发现不了的。流水线确保所有问题在到达 reviewer 之前已经被发现或修复。
