"no-mistakes 的流水线不是一键运行的 CI 黑盒。每个阶段都会产生 findings——机械问题自动修复，涉及意图的问题等你确认。只有全部通过，它才会说：'去创建 PR 吧。'"
01Pipeline 详解
当你的代码到达 no-mistakes 门禁后，七阶段流水线依次执行。每一阶段都可能产生 findings——分为两类：
⚡ auto-fix
机械问题
自动应用修复
🔍 ask-user
涉及意图
等待你审批
阶段 1 · review — AI 代码审查
AI 审查代码变更，检查逻辑错误、安全漏洞、性能问题、设计缺陷。

auto-findings：格式化、重命名、小重构
ask-user findings：设计决策、API 变更、安全影响
阶段 2 · test — 运行测试
在 disposable worktree 中运行测试套件。

auto-findings：测试用例中的小错误
ask-user findings：测试架构变更、mock 策略调整
阶段 3 · docs — 文档检查
检查是否需要更新文档、注释是否与实现一致。

auto-findings：自动补全缺失的注释
ask-user findings：API 文档变更、README 更新
阶段 4 · lint — 代码规范检查
运行 linter 检查代码风格和潜在问题。

auto-findings：自动格式化、修复 lint 错误
ask-user findings：（通常不会产生，lint 问题都可自动修复）
📌 注意：阶段 5-7（push → PR → CI）是自动接续的。只有前四个阶段全部通过（findings 全数处理），流水线才会执行 push 将代码转发到配置的远程仓库，PR 自动创建 Pull Request，CI 监控 CI 运行结果并在失败时自动修复。
02TUI 工作流
运行 no-mistakes 启动交互式 TUI（终端界面），这是最直观的使用方式。完整流程：
1
运行 no-mistakes
在项目目录中执行。Wizard 引导你选择分支、提交变更（未提交的文件自动处理）、推送到门禁。
2
流水线运行
TUI 实时显示各阶段进度。每个阶段完成时，findings 汇总到界面上。auto-fix 的自动应用，ask-user 的等待你处理。
3
处理 Findings
你已经看到所有问题了。对 auto-fix 的可以直接通过；对 ask-user 的你需要决定：approve（批准）、fix（手动修改）、skip（跳过此条）。
4
自动产出 PR
全部通过后，流水线自动 push 到远程、创建 PR、监控 CI。你不用离开终端——PR 链接会直接显示在 TUI 中。
📌 快捷操作：no-mistakes -y 全自动模式——跳过所有交互，auto-fix 自动应用，ask-user 的 finding 自动打印摘要。适合 CI 集成或自信满满的开发者。
03Agent Skill 集成
no-mistakes 最强的特性之一是 Agent 原生支持。初始化时安装的 /no-mistakes skill 让 AI 编程助手可以直接调用门禁。
💬 场景 1：先做任务，再门禁
你在 Claude Code 中说：
"帮我给用户列表加上分页功能，然后 /no-mistakes"

Claude 先写代码，然后自动触发门禁——review、test、docs、lint 一条龙。发现问题直接修复或问你。最终产出的是一个已经通过 AI 审查的 PR。
💬 场景 2：对已有代码跑门禁
你有已经改好的代码，直接键入：
/no-mistakes

Agent 会自动检查当前变更，触发门禁流水线。不需要手动 git push，不需要开 TUI——在对话中完成所有操作。
📌 底层原理：/no-mistakes 调用的是 no-mistakes axi——一个非交互式 TOON 接口。它专为 Agent 对话流程设计，不需要终端交互，所有输出通过标准输出返回给 Agent。
