今年 6 月，AI 编程圈出了件"离谱又合理"的事。

"我把 IDE 删了。我不再一行一行写提示词了。我设计了一堆循环，这些循环会自动给 Claude 分配任务、检查结果、决定下一步。我的工作从'写提示词'变成了'写循环'。"
这话一出，硅谷炸了。

02一句话解释：就是把"人做循环"变成"机器做循环"
如果你用过任何 AI 编程助手，你做的事长这样：
旧
Prompt Engineering 模式
你写指令 → AI 执行 → 你盯着看结果 → 不咋地对吧 → 你改指令 → AI 重新执行 → 你再盯 → 差不多了 → 你手动切到下一个任务。你，就是那个循环本身。
新
Loop Engineering 模式
你定好规则（什么时候触发、怎么算做完、做不完怎么办）→ 系统自动触发 AI → AI 干活 → 系统自动检查结果 → 不通过就自动修复重来 → 通过了自动切到下一步 → 循环，直到全部完成或触发预算上限。你，是定规矩的那个人。
翻译成大白话：
Prompt Engineering = 你是司机，AI 是车。你握着方向盘，踩油门，看路，转弯，停车。
Loop Engineering = 你是调度中心。你设好导航目的地、限速规则、检测点。车自己开，自己加油，到岔路口自己判断，堵车了自己绕——你只需要在它搞不定的时候接一下。
其实 Loop Engineering 不是突然冒出来的，它是 AI 工程化四层演进里最新的一层：
L1 · Prompt Engineering（2023）
"怎么措辞 AI 才听话？"——优化单次对话质量。核心是人的语言技巧。
L2 · Context Engineering（2024）
"给 AI 喂什么？"——RAG、MCP 工具、项目规则、上下文窗口管理。
L3 · Harness Engineering（2025）
"AI 怎么安全可靠执行？"——Hooks、权限控制、沙箱、验证、可观测性。ECC 就是这一层的代表作。
L4 · Loop Engineering（2026）
"AI 怎么持续自主推进？"——触发 → 执行 → 验证 → 记录 → 自动决策：继续 / 修复 / 停止 / 求救。
这四层是嵌套关系，不是替代关系。Prompt 没消失，它被 Loop 自动生成了。
03一个 Loop 长啥样？拆开给你看
Addy Osmani 把 Loop 拆成了六个零件，缺一个都跑不顺：
🫀
Automations — 心跳
定时或事件触发。比如每天早上 9 点扫一遍 GitHub Issues，有新 Bug 标签就自动拉起修复循环。也可以是你手动敲一个 /loop-start。
📦
Worktrees — 隔间
Git worktree 隔离。三个 Agent 并行干活，各占一个隔间，互不踩脚。谁改谁的，不打架。
📝
Skills — 经验库
项目的编码规范、架构决策、踩过的坑，写成 SKILL.md。Agent 每次启动先读这个，不用重新"教育"它。等于给 Agent 装了一个"组织记忆"。
🔌
Plugins & Connectors — 手脚
MCP 连接外部工具：Issue 系统、数据库、Slack 通知。Loop 不能活在真空里，得能读写真实世界的数据。
👥
Sub-agents — 写查分离
这条是铁律：写的和查的不能是同一个人。一个 Agent 写代码，另一个 Agent（不同模型、不同指令）来审查。自己写的代码自己查，跟没查一样。
🧠
Memory — 记忆
跨对话持久化状态。Agent 本轮做了什么，哪些通过了，哪些还差着——写到文件里。下一轮 Agent 启动时读这个文件，无缝接上。不靠"上下文窗口"记东西，靠文件系统。
04设计 Loop 的七条铁律
GitHub 上的开源 Loop Engineering 技能文件总结了七条原则，每条都是用真金白银烧出来的教训：
❶
权力从 Prompt 移到了 Loop
别再憋超级提示词了。花精力设计控制流——触发条件、分支逻辑、退出条件。Loop 本身比 Loop 里面的单次提示词重要十倍。
❷
"完成"必须是机器能判定的
不能是"我觉得差不多了"。得是二进制条件：所有测试通过 + lint 零报错 + 覆盖率 ≥80%。主观判断是 Loop 最大的坑。
❸
验证靠工具，不靠 Agent 自报
永远不要问 Agent "你做好了吗？"——它一定会说做好了。跑测试、跑 lint、跑构建。Agent 的嘴不能信，只能信工具的返回值。
❹
上线前定义所有出口
成功的出口、失败的出口、预算烧光的出口、卡死不动（no-progress detection）的出口、升级到人工的出口。开放式循环等于烧钱机器。
❺
上下文是有限资源，别当自来水用
每轮裁剪输出、用压缩、记笔记到文件。子 Agent 是天然的上下文隔离器——烧的是它的上下文，不是你的主对话。
❻
文件系统就是记忆，每轮新鲜上下文启动
别指望 AI 的"长上下文"能记住一切。把做了什么、学到什么、还剩什么写到文件里。下一轮 Agent 打开文件看一眼，立刻续上。
❼
最难的不是自主运行，是验证、停止和求救
让 AI 自己跑不难，GPT-4 天生就会。难的是它什么时候该停下来，什么时候该说"我搞不定了"，以及你怎么知道它做的对不对。半自主（semi-autonomous）比全自主更靠谱——涉及不可逆操作（发 PR、部署），拦一下。
05放 Loop 之前先想清楚这 5 个问题
别激动，不是所有活儿都需要 Loop。一个简单的问答用 Loop 纯属浪费。但如果你准备上一个 Loop，先把这五个问题答清楚了：
Q1：怎么算成功？
必须是二进制的。"所有测试绿 + lint 零报错"。不能是"代码看起来不错"。
Q2：谁来验收？
必须独立于执行者。独立的验证 Agent + 对抗性审查 + 多维度评估。
Q3：什么状态要跨轮持久化？
已处理列表、已验证列表、失败尝试列表。不是全存，是存"决策所需的最小状态"。
Q4：什么时候停？
成功条件 + Token 预算上限 + N 轮无新发现则停（loop-until-dry）。三重保险。
Q5：什么时候拉人进来？
置信度过低、连续矛盾、涉及安全敏感操作（发 PR、部署、改数据库 Schema）。
06不是所有活儿都值得用 Loop——但也有个简单判断
别被大佬们的话吓到。不是说从明天开始你就不写提示词了。该写还是写。判断一个任务适不适合用 Loop 就一个标准：
这个任务需不需要你反复"盯 → 改 → 盯 → 改"？
不用 → 直接写提示词，一次搞定，不折腾。
需要 → 这就是 Loop 的用武之地。定义验收标准，让 AI 自己循环。
几个典型场景：
🔍
Bug 狩猎循环
定时扫 Issues → 发现新 Bug → 自动分给修复 Agent → 修完跑全量测试 → 通过则自动 PR → 没通过重新修 → 三轮还不过，@人工。loop-until-dry：连续两轮没发现新 Bug 就停。
🧹
代码质量巡逻循环
每天凌晨跑一轮：静态分析 → 安全扫描 → 依赖检查 → 发现问题自动建 Issue → 触发修复循环。项目质量是"跑"出来的，不是"喊"出来的。
📄
文档同步循环
检测到 API 签名变更 → 自动更新对应文档 → 检查示例代码是否还能跑 → 不能就修。
07Loop 工程的三个大坑，踩过的人都不说话
💰
Token 账单爆炸
一个 Loop 里有多个子 Agent 并行跑，每个都在消耗 Token。开放式循环（没有预算上限的）一晚上可能烧掉一个月的额度。一定要设 budget cap。
🧟
理解力负债
这是个新词，但特别真实。Loop 跑得越顺，产出的代码越多——你对产出的理解就越薄弱。一天跑 10 个 PR 出来，你真看过几个？如果出了问题，你能不能快速定位？这不是技术问题，是认知问题。
🎢
跳过 Harness 直接上 Loop = 灾难
如果你的单个 Agent 任务都不能可靠执行，自动化只会把不可靠复制 N 份、放大 N 倍。先把 Harness 的事做好（权限、验证、约束），再谈 Loop。
08所以，我现在该怎么上手？
别想一口吃个胖子，分三步走：
①
先把 Harness 搞好
如果你的 Claude Code / Cursor 还是"裸奔"状态——没装 ECC、没配 Rules、没设 Hooks——先去把这个弄好。Loop 跑在 Harness 上，地基不稳别盖楼。
②
从最小 Loop 开始
做一个"定时触发 + 单一验收条件（测试全绿）+ 简单验证 + 最多跑 3 轮"的小闭环。比如：提交代码 → 自动跑 lint → 不通过自动修 → 再跑 → 通过 / 3 轮不过则停。
③
读两个开源项目
GitHub 上搜 loop-engineering（maxmilian 那个），把七条原则和评估套件看一遍。再装个 ECC（224K Star），它的 autonomous-loops、loop-operator、/loop-start 命令是 Loop Engineering 的工程化实现。
